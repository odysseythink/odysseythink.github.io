Work Stealing (工作窃取) 动态演示

当一个 P (Processor) 的本地队列空了之后，它会尝试从其他 P 的本地队列中窃取 G 来执行。 “偷一半”是相对于被偷的那个 P 的本地队列而言的。 通常是从队尾偷取。

当前状态：P1 闲置，P2 繁忙

P1 的本地队列已经清空（0 个 G），而 P2 的本地队列积压了大量的 G。为了保证 CPU 利用率，P1 即将触发 Work Stealing 机制。

GC 算法与内存逃逸

三色标记清扫算法 (Tri-color Mark & Sweep)

Go 采用并发的三色标记清扫机制。其基本原理是将对象分为黑、灰、白三色：

  • 黑色：该对象及其引用的所有子对象都已被 GC 扫描完毕。存活对象。
  • 灰色：该对象已被扫描，但其引用的子对象还未全部扫描。作为波面(Wavefront)推进。
  • 白色：未被扫描到的对象。如果在标记阶段结束后仍为白色，将被作为垃圾回收。

写屏障 (Write Barrier)

因为 Go 是并发 GC，用户态代码 (Mutator) 可能会在标记过程中修改引用关系，导致存活对象被错误清理。为了解决此问题，Go 引入了写屏障机制。

混合写屏障 (Hybrid Write Barrier): Go 1.8 引入，结合了插入写屏障和删除写屏障的优点。栈上不开启屏障以保证性能，仅堆上开启。当一个堆对象指针被修改时，屏障逻辑会被触发，将相关对象着色为灰色，保证可达对象不会丢失。

什么是内存逃逸？ (Memory Escape)

在 Go 中，变量不仅可以分配在栈 (Stack) 上，也可以分配在堆 (Heap) 上。栈上的内存由编译器自动管理，函数返回时自动回收，开销极小。而堆上的内存需要 GC 介入回收，开销较大。

内存逃逸 指的是编译器在编译阶段经过逃逸分析后，发现本该分配在栈上的局部变量，因为生命周期或作用域超出了函数范围，不得已将其分配到了堆上。

场景 1: 返回局部变量指针

函数内创建的局部变量，其地址（指针）作为返回值传递给外部。外部函数可能会继续使用该地址，因此变量生命周期延长，必须分配在堆上。

场景 2: 发送到 Channel

将局部变量的指针或包含指针的结构体发送到 Channel 中。因为在编译时无法预知 Channel 的接收方何时执行，导致生命周期不可控。

场景 3: 接口动态分派 / interface{}

例如调用 fmt.Println(x)，因为函数的参数是 interface{}，在编译期很难确定其具体的类型和大小，经常会导致逃逸。

场景 4: 闭包引用

函数内定义的匿名函数（闭包）使用了外部函数的局部变量。由于闭包本身可能在外部执行，这些被引用的变量也会逃逸到堆上。

场景 5: 切片扩容或过大

栈空间是有限的。如果定义的切片/数组过大，或者在运行时切片发生了扩容且大小超过了编译期计算的范围，也会分配到堆上。
